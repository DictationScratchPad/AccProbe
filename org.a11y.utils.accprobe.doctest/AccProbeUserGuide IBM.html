Introduction

The Accessibility Probe (AccProbe) can be used to perform the following tasks.

  • Viewing the Microsoft Active Accessibility (MSAA) or IAccessible2 hierarchy of a currently running application.
  • Examining the accessibility properties of the GUI controls of each application.
  • Monitoring the events generated by the GUI controls.

AccProbe is useful for verifying that the MSAA or IAccessible2 properties of a GUI control have been coded properly, and for debugging problems found when testing an application user interface with a screen reader or other assistive technology.

AccProbe combines the functionality found in tools like Microsoft Inspect32, AccExplorer, and AccEvent, into one tool which works for both MSAA and IAccessible2 objects.

AccProbe is an Eclipse Rich Client Platform (RCP) application and is available as part of the Eclipse Accessibility Tools Framework (ACTF). (However, AccProbe is a standalone application, and does not require the use of the Eclipse development framework.)


Download and Installation

AccProbe is provided as a zip file which contains all the components needed for it to run as a standalone Windows application in conjunction with the appropriate Java Runtime Environment. Installation instructions are as follows.

Obtaining the Required Java Runtime Environment

AccProbe currently requires a Java Runtime Environment (JRE) version 1.5.x or 1.6.x installation. The JRE must be obtained from the IBM Hursley web site described below.

If a JRE has been previously installed, check the version by opening a Command Prompt window (Start->All Programs->Accessories->Command Prompt) and invoking the command "java -version".

  • If this command returns a version of 1.5.x or 1.6.x, proceed to the next section for Installing AccProbe.
  • If this command returns a version other than 1.5.x (for example 1.4.2), download and install Java 5 or Java 6 as follows:
    • Go to IBM Hursley Java Installation Manager (JIM) site.
    • In the How to download code... section, select Access JIM.
    • In the Finding things in JIM section, select the Windows (Intel 32-bit) link .
    • In the Releases section, select 5.0 Service Release xxx for Java 5 or select 6.0 Service Release yyy for Java 6.
    • In the Binary Downloads section, select the JRE Package (InstallShield) link and log in with a valid IBM Intranet user ID and password.
    • Select the "download is for personal use" link and save the resulting file.
    • Execute the file to install the runtime environment.

Installing AccProbe

AccProbe is provided as a zip file and is installed as follows:

  • Create a directory which will be used to contain the AccProbe files.
  • Download the AcccProbe zip file into the directory created above.
  • Extract the files from the zip file.

IAccessible2 Installation Considerations

To use AccProbe with the IAccessible2 interface, the IAccessible2Proxy.dll must be registered. Two methods to do this are listed below. (Use the correct path to the folder in which AccProbe was installed.)

  • From a command prompt, type "regsvr32 'path to accprobe directory'\IAccessible2Proxy.dll"
  • Or, from the start menu, choose Run. Type "regsvr32 'path to accprobe directory'\IAccessible2Proxy.dll" in the input field and press the OK button.

(If regsvr32 is not in your path, you may need to use the full path. Usually this is c:\windows\system32\regsvr32).

Using AccProbe to Debug Screen Reader Problems

AccProbe may be used primarily for two main purposes:

  1. As a development aid, to verify the accessibility properties and events are being set properly for user interface elements.
  2. As a debug tool, to debug problems found when testing with a screen reader, or other assistive technology.

This section discusses some of the issues to consider when debugging a screen reader problem.

Reasons a Screen Reader Might Fail to Announce Information as Expected

Screen readers and other assistive technologies are used to test whether a product is accessible. At times, problems are found while navigating the user interface and the screen reader does not announce information as expected. Here are some possible reasons a screen reader might fail to announce expected information.

  1. The accessibility properties (MSAA or IAccessible2) are not coded properly.
  2. The accessibility events are not being triggered properly.
  3. The screen reader has a problem.
These issues are discussed in more detail below.

Improperly Coded Accessibility Properties (MSAA or IAccessible2)

The screen reader uses, among other things, the properties provided through the accessibility interface to determine what to announce. If the accessible properties are not set properly, the screen reader may not know what to announce.

Here are some of the properties commonly used by the screen reader when deciding what to announce.

  • name
  • value
  • role
  • state
  • other properties, depending on the type of control.

Improperly Coded Accessibility Events

The screen reader monitors certain events which trigger the screen reader's spoken output. If those events are not triggered by the application, the screen reader may be unaware that spoken output is required.

Here are some of the events most like used by the screen reader when deciding when to announce more information.

  • focus change,
  • state change,
  • value change,
  • selection change,
  • other events, depending on the type of control.

Screen Reader Problems

Since a screen reader is a software application, it may also have coding errors. If the control in an application is coded properly for all the accessibility properties and events but the screen reader does not announce it correctly, the screen reader's support team might be contacted to help resolve the problem.

Strategy for Debugging Screen Reader Problems with AccProbe

If the screen reader does not announce the expected information for a user interface element, here is a suggested strategy for determining where the problem lies.

  1. Start AccProbe and the application whose interface is not working properly with the assistive technology (screen reader).
  2. Position the application window and AccProbe window so both can be viewed at the same time, if possible.
  3. Use the AccProbe Explorer View and the Accessibility Properties View to display the accessibility properties (either MSAA or IAccessible2, as appropriate), for the failing user interface element. Then, either drill down through the hierarchy in the Explorer View, or use the tracking options described in the AccProbe Viewing and Tracking Options section to navigate to the element of interest in the application.
  4. Verify that the properties are as expected. For hints on which properties are of interest, see the discussion of Improperly Coded Accessibility Properties.
  5. Use the AccProbe Event Monitor View to verify the accessibility events are being triggered as expected. For hints on which events are of interest, see the discussion of Improperly Coded Accessibility Events.
